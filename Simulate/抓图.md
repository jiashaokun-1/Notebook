
### 核心抽象
- `base.py` — `GraphExtractor` 抽象基类，定义 `extract()` 接口
- `op_registry.py` — `OpRegistry` 操作注册表，将不同框架的算子名映射到统一的 `OpType`

### 提取器（5 种来源）
1. **`torch_extractor.py`** — 通过 `torch.fx` 符号追踪 PyTorch 模型提取计算图
2. **`export_extractor.py`** — 通过 `torch.export` 导出 PyTorch 2.x 模型提取计算图
3. **`dispatch_extractor.py`** — 通过 `TorchDispatchMode` 拦截 aten 算子（支持动态控制流、MoE 路由、自定义 attention），在 meta tensor 上执行并记录算子
4. **`onnx_extractor.py`** — 从 ONNX 模型文件提取计算图
5. **`profiler_extractor.py`** — 从 PyTorch profiler 的 Chrome trace JSON 提取计算图

### 配置驱动构建
- **`config_extractor.py`** — 从 HuggingFace `config.json` 构建计算图，支持多种架构：
  - `StandardTransformerHandler` — LLaMA、Mistral、Qwen2、Mixtral、GPT-2、Falcon 等
  - `DeepSeekHandler` — DeepSeek-V2/V3（MLA attention + 混合 dense/MoE）
- **`config_normalizer.py`** — 将不同 HF 架构的 config 字段归一化为统一的 `ModelConfig`
- **`graph_builder.py`** — 手动构建计算图的 DSL，提供 matmul、linear、attention、MoE 等原语和复合块，支持 TP/EP 并行切分


根据代码分析，`frontend` 目录实现了 **5 种计算图提取器**，它们从不同来源将模型转换为 `ComputeGraph`：

---

## 1. `TorchGraphExtractor` — FX 符号追踪

**原理**：使用 `torch.fx.symbolic_trace()` 对 PyTorch 模型进行静态追踪

```python
extractor = TorchGraphExtractor()
graph = extractor.extract(model, (input_tensor,))
```

**特点**：
- 通过 `torch.fx.Interpreter` 运行一次模型捕获中间 tensor shape
- 映射 `nn.Module` 类型（如 `nn.Linear` → `aten::linear`）和 `torch.nn.functional` 调用
- **局限**：无法处理动态控制流（if/else 依赖 tensor 值）、MoE 路由等

**适用场景**：简单静态模型（MLP、CNN）

---

## 2. `ExportExtractor` — torch.export 导出

**原理**：使用 PyTorch 2.1+ 的 `torch.export.export()` 导出 FX 图

```python
extractor = ExportExtractor()
graph = extractor.extract(model, (input_tensor,))
```

**特点**：
- 比 FX 追踪更严格，支持更多控制流（通过 AOTAutograd）
- 从 `fx_node.meta["val"]` 获取 tensor shape/dtype
- 能捕获 `get_attr` 节点（嵌入的权重参数）
- **局限**：仍需静态图结构，动态分支支持有限

**适用场景**：需要更准确图表示的 PyTorch 2.x 模型

---

## 3. `DispatchExtractor` — TorchDispatchMode 拦截 ⭐

**原理**：继承 `TorchDispatchMode`，在模型**实际执行时**拦截每一个 aten 算子

```python
extractor = DispatchExtractor(skip_reshapes=True)
graph = extractor.extract(model, {"input_ids": input_ids})

# 或直接从 HF 模型 ID 提取（meta device 上运行，无需 GPU）
graph = extractor.extract_from_config("deepseek-ai/DeepSeek-V3", num_hidden_layers=2)
```

**特点**：
- **最强大的提取器**：支持动态控制流、MoE 路由、自定义 attention
- 通过 `_ModuleTracker`（forward pre/post hooks）追踪算子所属模块路径（如 `L3.self_attn.q_a_proj`）
- 自动将模型移到 **meta device** 执行（只做 shape 传播，无实际计算）
- 捕获极细粒度 aten op（DeepSeek V3 约 6500+ 个算子）
- 需要配合 `DISPATCH_FUSION_RULES` 融合后才能评估
- 通过 `data_ptr()` 追踪 tensor 数据依赖关系构建边

**适用场景**：任何 PyTorch 模型，尤其是复杂 LLM（DeepSeek、MoE）

---

## 4. `ONNXExtractor` — ONNX 模型解析

**原理**：解析 ONNX `ModelProto`，遍历 `graph.node` 构建计算图

```python
extractor = ONNXExtractor()
graph = extractor.extract("model.onnx")
```

**特点**：
- 依赖可选的 `onnx` 包（`pip install 'xpu-simulator[onnx]'`）
- 自动执行 `onnx.shape_inference.infer_shapes()`
- 从 `initializer` 获取权重 shape，从 `value_info`/`output` 获取中间 tensor shape
- 跳过纯常量节点（输入全是 initializers 的节点）
- 通过 `tensor_producer` 映射建立数据依赖边

**适用场景**：已导出为 ONNX 格式的模型

---

## 5. `ProfilerExtractor` — Profiler Trace 重建

**原理**：从 PyTorch profiler 输出的 Chrome trace JSON 重建计算图

```python
extractor = ProfilerExtractor()
graph = extractor.extract("trace.json")
```

**特点**：
- 解析 Chrome trace 格式（`traceEvents` 数组，`ph="X"` 的完整事件）
- 过滤计算相关类别（`cpu_op`, `kernel`, `operator`）
- 保留 profiler 属性：`profiled_latency_us`、`profiled_start_us`
- **边推断是 best-effort**：通过匹配前后算子的 input/output shape 来推断依赖关系
- 过滤掉 < 0.1µs 的框架开销事件

**适用场景**：已有真实运行 trace，想基于实测数据重建图结构

---

## 对比总结

| 提取器 | 来源 | 粒度 | 动态控制流 | 需要实际模型 | 典型算子数(8B) |
|--------|------|------|-----------|-------------|---------------|
| `TorchGraphExtractor` | FX trace | 粗（module 级） | ❌ | ✅ | ~200 |
| `ExportExtractor` | torch.export | 中 | 部分 | ✅ | ~300 |
| `DispatchExtractor` | TorchDispatchMode | **极细（aten 级）** | ✅ | ✅（但用 meta device） | ~1800 |
| `ONNXExtractor` | ONNX 文件 | 中 | ❌ | ❌（只需文件） | ~300 |
| `ProfilerExtractor` | Chrome trace JSON | 细 | ✅（已执行过） | ❌（只需 trace） | ~500 |

**核心差异**：
- **ConfigExtractor**（第 6 种，配置驱动）从 `config.json` **逻辑构建**图，不依赖实际模型代码
- **DispatchExtractor** 是唯一能处理 MoE 动态路由、条件分支的提取器
- **ProfilerExtractor** 是唯一能携带实测延迟数据的提取器
